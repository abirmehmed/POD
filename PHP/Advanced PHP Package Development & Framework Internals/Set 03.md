# Advanced PHP Package Development & Framework Internals: Set 3 (Final)
## Automated Refactoring, Static Analysis & Mutation Testing

## 📘 10 Conceptual Questions
1. **PHPStan Custom Rules:** How do you write a custom PHPStan rule? Why is it vastly superior to regex-based linters (like `grep` or simple IDE checks) for enforcing domain-specific architectural boundaries (e.g., "Controllers must never instantiate Eloquent models directly")?
2. **Rector & Abstract Syntax Trees (AST):** How does Rector parse PHP code into an AST and apply transformations? Why is this method infinitely safer than search-and-replace (regex) for major framework upgrades (e.g., migrating Laravel 10 syntax to Laravel 11)?
3. **Mutation Testing (Infection) Reality Check:** What is mutation testing, and why is "100% code coverage" a dangerous lie? Explain how tools like Infection mutate code (e.g., changing `>` to `>=`, or `true` to `false`) to prove that your tests *actually* catch bugs, rather than just executing the code.
4. **Architectural Boundary Enforcement:** How do tools like `phparkitect` or `deptrac` enforce layer separation (e.g., "Domain layer must not depend on Infrastructure") at the CI level? What is the performance cost of running these checks on a large codebase?
5. **PHPDoc vs. Native Types:** Why is relying solely on PHPDoc for type hinting a vulnerability in modern PHP? How does PHPStan’s strictness level (up to 10) force the adoption of native union/intersection types, `readonly` properties, and generics?
6. **Generics in PHP:** How do PHPStan/Psalm annotations (e.g., `@template`, `@param Collection<User>`) bring true generic type safety to PHP? Why is this crucial for package development when returning collections or repositories?
7. **The Baseline Strategy:** What is a PHPStan/Rector baseline file, and how does it enable incremental adoption of strict static analysis in a legacy codebase without causing CI to fail immediately on day one?
8. **Dead Code Detection:** How do static analysis tools identify unused methods, properties, and classes? Why is aggressively removing dead code critical for package maintainability, security, and reducing cognitive load?
9. **CI/CD Quality Gates:** How do you configure a CI pipeline to run PHPStan, Rector (dry-run), and Infection in parallel? Why should these checks be non-negotiable blockers for merging a Pull Request?
10. **The Cost of Over-Engineering:** When does enforcing PHPStan Level 10 or 100% mutation coverage become counterproductive? How do you balance absolute strictness with developer velocity and pragmatic package development?
