# Rust Programming Language: Set 3
## Fearless Concurrency & Async Rust

## 📘 10 Conceptual Questions
1. **Threads vs. Processes:** How does Rust's `std::thread::spawn` map to OS-level threads? Why is creating thousands of OS threads inefficient, and how does this limitation lead to the need for asynchronous programming?
2. **Message Passing (`mpsc`) vs. Shared State:** What are the trade-offs between communicating via channels (e.g., `std::sync::mpsc`) versus sharing memory via `Arc<Mutex<T>>`? Why is the Go proverb "Do not communicate by sharing memory; instead, share memory by communicating" highly relevant in Rust?
3. **The `Send` and `Sync` Marker Traits:** These are the foundational pillars of Rust's concurrency safety. What does it mean for a type to be `Send` (safe to transfer ownership across threads) vs. `Sync` (safe to share references across threads)? Why are raw pointers (`*const T`) neither?
4. **`Rc<T>` vs. `Arc<T>`:** Why is the standard `Rc<T>` (Reference Counted) smart pointer not thread-safe? How does `Arc<T>` (Atomic Reference Counted) use atomic CPU instructions to safely manage reference counts across multiple threads, and what is the performance cost of this?
5. **`Mutex<T>` and Lock Poisoning:** How does a `Mutex` provide interior mutability across threads? What is "lock poisoning" (when a thread panics while holding a lock), and how does Rust force you to handle the `PoisonError` to prevent using potentially corrupted data?
6. **The `Future` Trait & Lazy Evaluation:** In Rust, an `async` function doesn't execute immediately; it returns a `Future`. What is a `Future` under the hood (a state machine), and why must it be `.await`ed or spawned to make progress?
7. **The Async Runtime (Tokio):** Why does Rust's standard library *not* include an async executor? How does a runtime like Tokio use a work-stealing thread pool to poll thousands of Futures concurrently on a handful of OS threads?
8. **The `Send` Bound Across `.await` Points:** Why does the compiler throw an error if you hold a `MutexGuard` (or any non-`Send` type) across an `.await` point? How do you refactor code to drop the guard *before* the `.await`?
9. **Blocking the Async Runtime:** Why is calling a synchronous, blocking function (like `std::fs::read` or a heavy CPU calculation) inside an `async` function a cardinal sin in Tokio? How does `tokio::task::spawn_blocking` solve this?
10. **Concurrency vs. Parallelism:** How does asynchronous Rust (I/O concurrency) differ from data parallelism? Explain how the `rayon` crate uses thread pools and work-stealing to parallelize CPU-bound tasks (like image processing) across all available cores.
