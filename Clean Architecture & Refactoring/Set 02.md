# Clean Architecture & Refactoring: Set 2
## Advanced Patterns, Anti-Patterns & Test-Driven Transformation

## 📘 10 Conceptual Questions
1. **Strangler Fig Pattern:** How do you safely replace legacy code with new architecture without a big-bang rewrite? Explain incremental routing, dual-write/dual-read strategies, and when to finally cut over.
2. **Replace Conditional with Polymorphism:** Why are long `if/else` or `switch` chains a maintenance trap? How do you refactor them into strategy/state patterns without scattering logic or creating class explosion?
3. **Anemic vs. Rich Domain Models:** What makes a model "anemic" (getters/setters only)? How do you move behavior into entities/value objects while keeping Eloquent's convenience from corrupting domain purity?
4. **Service Locator Anti-Pattern:** Why is calling `app()->make()`, `resolve()`, or `Container::getInstance()` inside business logic considered harmful? How does it hide dependencies, break testability, and violate Dependency Inversion?
5. **Test Doubles Strategy:** When should you use Fakes, Stubs, Mocks, or Spies in clean architecture tests? How do you avoid over-mocking and test only the contract, not the implementation details?
6. **Circular Dependencies & Decoupling:** How do you detect and break circular references between modules or classes? Explain dependency inversion, event-driven decoupling, and interface extraction as remediation strategies.
7. **Primitive Obsession:** Why is passing raw strings/ints for concepts like `Email`, `Money`, `Status`, or `PhoneNumber` dangerous? How do custom value objects or backed enums improve validation, type safety, and domain clarity?
8. **Refactoring to Patterns:** When should you recognize that code naturally fits a known pattern (Observer, Strategy, Factory, Command)? How do you apply them incrementally without over-engineering upfront?
9. **State vs. Behavior Separation:** How do you decide what belongs in a DTO/struct vs a domain entity vs a service? What breaks when you mix data carriers with business logic or leak infrastructure concerns into the domain?
10. **The Boy Scout Rule & Continuous Refactoring:** Why is "leave the code better than you found it" critical in large codebases? How do you balance feature delivery with technical debt payoff without blocking releases or creating merge hell?

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Extract Strategy Pattern**<br>Find a method with 4+ `if/else` branches (e.g., discount calculation, shipping method routing). Refactor into a `ShippingStrategy` interface + concrete implementations. Container resolves at composition root only. | Conditional chain eliminated. Each strategy is isolated & testable. New strategy added without modifying existing code. OCP verified. |
| 2 | **Eliminate Primitive Obsession**<br>Replace raw `string $email`, `float $price`, `string $status` with `EmailAddress`, `Money`, `OrderStatus` value objects/enums. Update forms, DB casts, validation. Verify type safety across layers. | Zero raw primitives for domain concepts. IDE autocomplete & static analysis catch invalid values. Validation lives in VO constructor/factory. |
| 3 | **Break Circular Dependency**<br>Identify two classes that depend on each other. Extract shared interface or introduce domain event to decouple. Run static analysis to verify zero circular references. Both remain independently testable. | Dependency graph shows no cycles. Communication happens via abstraction or event. Tests pass without bootstrapping both classes together. |
| 4 | **Strangler Fig Migration**<br>Pick a legacy controller/route. Create new clean architecture endpoint alongside it. Route 10% of traffic via feature flag or middleware. Gradually shift to 100%. Remove old code. Document cutover. | Both old & new endpoints coexist safely. Traffic shifts incrementally. Zero downtime or data inconsistency. Legacy code deleted after verification. |
| 5 | **Rich Domain Model Extraction**<br>Take an Eloquent model with business logic scattered in controllers. Move rules (e.g., `canBeShipped()`, `calculateTax()`, `applyDiscount()`) into the model or dedicated domain service. Remove procedural checks from controllers. | Controllers only orchestrate. Model/services contain behavior. Tests verify business rules without HTTP layer. No `if ($status == 'x')` outside domain. |
| 6 | **Fake Implementation Testing**<br>Write a unit test for a service that depends on an external API. Create a `FakeApiService` implementing the same interface, returning canned data. Verify service logic without network calls or mocks. | Test runs offline in <10ms. Fake mirrors real contract. No `Mockery`/`PHPUnit` mocks used. Service behavior verified purely via inputs/outputs. |
| 7 | **Composition Root & DI Registration**<br>Find `app()->make()` or `resolve()` in business logic. Replace with constructor injection. Register all bindings in `AppServiceProvider` or dedicated module provider. Verify container resolves graph correctly. | Zero container calls outside composition root. Dependencies explicit in constructors. `php artisan tinker` can resolve full graph. Tests swap implementations easily. |
| 8 | **Extract Command/Query Objects**<br>Split a fat method that reads and writes data into `GetProductInventoryQuery` and `UpdateStockCommand`. Each has single responsibility, explicit inputs/outputs, and no side-effect mixing. | Query returns data only. Command performs state change only. Controllers route to them. Each is unit-testable in isolation. |
| 9 | **Refactor God Object**<br>Take a class with >15 public methods or mixed concerns (e.g., `UserManager` handling auth, profile, billing, notifications). Split into focused services. Update consumers incrementally without breaking existing flows. | Original class shrinks >60%. New services have single responsibility. Tests verify each concern independently. No behavior regression. |
| 10| **Architecture Enforcement Tests**<br>Write PHPUnit tests that assert architectural rules: e.g., `Domain` never imports `Illuminate\`, `Infrastructure` depends on `Domain` but not vice versa, Controllers only depend on Services/Actions. Fail CI on violation. | Tests pass on clean code, fail on architectural breach. Rules documented. CI blocks non-compliant merges. Team adopts as quality gate. |
