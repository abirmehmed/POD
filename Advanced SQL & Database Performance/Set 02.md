# Advanced SQL & Database Performance: Set 2
## Advanced Schema Design, Anti-Patterns & Data Modeling

## 📘 10 Conceptual Questions
1. **Normalization vs. Denormalization:** Third Normal Form (3NF) eliminates data redundancy, but why do high-performance systems intentionally denormalize data? Explain the trade-off between write amplification and read performance, and how to maintain consistency in denormalized tables.
2. **JSON/JSONB Columns vs. Relational Tables:** When should you store semi-structured data in a JSON column versus creating a separate relational table? What are the performance implications of querying JSON data, and how do modern databases (PostgreSQL JSONB, MySQL 8.0) index it?
3. **The Entity-Attribute-Value (EAV) Anti-Pattern:** Why is EAV considered a design flaw in relational databases? How does it destroy type safety, complicate querying, and kill performance? What are the modern alternatives (e.g., JSON columns, sparse columns)?
4. **Polymorphic Relationships at the DB Level:** Laravel supports polymorphic relations easily, but why are they problematic in a strict relational database? Explain the loss of foreign key constraints, referential integrity, and how to enforce integrity at the database level.
5. **Surrogate Keys vs. Natural Keys (UUIDs vs. Auto-Incrementing INTs):** What are the performance implications of using UUIDs as Primary Keys compared to auto-incrementing integers? Explain index fragmentation, write amplification, and how B-Tree structures handle sequential vs. random inserts.
6. **Soft Deletes & Unique Constraints:** Why does implementing "soft deletes" (`deleted_at`) complicate unique constraints? How do you enforce uniqueness for active records while allowing the same value for soft-deleted records in the database?
7. **Date and Time Storage:** What is the difference between `TIMESTAMP`, `DATETIME`, and `DATE`? How do timezones affect database storage, and why is storing everything in UTC and converting at the application layer the standard practice?
8. **Handling Large Text and BLOBs:** How do databases handle massive text fields (e.g., PostgreSQL's TOAST, MySQL's `TEXT`/`BLOB`)? Why should you generally avoid storing large files or huge text blocks directly in the database, and when is it actually acceptable?
9. **Materialized Views for Read-Heavy Workloads:** What is a Materialized View, and how does it differ from a standard View? When should you use it to cache complex aggregations or joins, and how do you handle data staleness (refresh strategies)?
10. **Schema Evolution without Downtime:** How do you safely rename a column, change a data type, or split a table in a production database with zero downtime? Explain the "Expand and Contract" pattern in the context of schema changes.
