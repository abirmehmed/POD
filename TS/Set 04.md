# TypeScript Programming Language: Set 4
## The Ecosystem: Frameworks, Tooling, Testing, and Production Architecture

## 📘 10 Conceptual Questions

1. **TypeScript in React Server Components (RSC):** How does TypeScript typing differ between React Server Components and Client Components? Why can't you pass functions or client-side state as `props` to a Server Component, and how do you type `async` component functions that fetch data directly?
2. **Testing Types vs. Testing Runtime:** When testing TypeScript code (using Vitest or Jest), what is the difference between testing *runtime behavior* and testing *compile-time types*? How do utilities like Vitest's `expectTypeOf` or the `tsd` library allow you to write unit tests that fail if a type changes, even if the runtime code works?
3. **Monorepos & Project References:** How do TypeScript Project References (`composite: true`, `references: []`) work under the hood? Why are they absolutely essential for scaling TypeScript in a monorepo (using Turborepo or Nx), and how do they enable incremental compilation to speed up massive codebases?
4. **Strict `try/catch` & The `unknown` Catch Variable:** In strict mode, TypeScript types the caught variable in a `catch (e)` block as `unknown` instead of `any` or `Error`. Why did the TS team make this change? How do you safely narrow `unknown` to a custom `AppError` type to extract properties like `statusCode` without using `as` casts?
5. **OpenAPI/Swagger & End-to-End Type Safety:** How do you generate TypeScript types directly from a backend's OpenAPI/Swagger specification? Why is this considered the "holy grail" of frontend-backend type safety, and how do tools like `openapi-typescript` or `orval` automate the creation of fully typed API clients?
6. **Typing CSS Modules & Tailwind:** TypeScript doesn't natively understand CSS. How do you get strict type safety for CSS class names? Explain how tools like `vite-plugin-css-modules-types` or `tailwind-variants` bridge the gap, ensuring that `styles.nonExistentClass` throws a compile-time error.
7. **`tsconfig.json` `paths` vs. Runtime Resolution:** How does the `paths` alias feature in `tsconfig.json` (e.g., mapping `@/*` to `./src/*`) work? Why does it *only* work for the TypeScript compiler and bundlers, and why will your Node.js app crash at runtime if you don't configure a runtime resolver (like `tsx` or `tsconfig-paths`)?
8. **AsyncGenerators & Iterator Types:** How do you explicitly type an asynchronous generator function? Explain the three generic parameters of `AsyncGenerator<YieldType, ReturnType, NextType>` and how they map to the `yield`, `return`, and `next()` methods.
9. **TypeScript Build Performance & `isolatedModules`:** Why do modern bundlers (Vite/esbuild/SWC) require `"isolatedModules": true` in your `tsconfig.json`? How does this restriction prevent you from using certain TypeScript features (like `const enum` or type-only re-exports) to ensure each file can be transpiled independently in milliseconds?
10. **Typing Complex Third-Party JS Libraries:** When you install a legacy JavaScript library that lacks `@types/` definitions, how do you write a comprehensive ambient declaration (`.d.ts`) for it? How do you type overloaded functions, optional parameters, and global namespaces so your IDE provides perfect IntelliSense?
