# TypeScript Programming Language: Set 1
## The TypeScript Paradigm Shift: Static Typing, Generics, and the Type System

## 📘 10 Conceptual Questions

1. **Structural vs. Nominal Typing:** In languages like Java or C#, if two classes have the exact same properties but different names, they are incompatible (Nominal typing). TypeScript uses **Structural Typing** (Duck Typing). What does this mean? If a `Dog` interface and a `Wolf` interface have the exact same properties, can you pass a `Dog` where a `Wolf` is expected? Why?
2. **`interface` vs. `type` Aliases:** Both can define object shapes, but they have distinct differences. How do they differ regarding **Declaration Merging** (extending interfaces with the same name) and **Union/Intersection capabilities**? When should you strictly use one over the other in modern TS?
3. **The Type Hierarchy: `any` vs. `unknown` vs. `never`:** What is the fundamental difference between these three special types? Why is `any` considered dangerous and an "escape hatch"? How does `unknown` force you to perform type narrowing before using a value, and when does a type resolve to `never` (e.g., impossible intersections or exhaustive checks)?
4. **Union (`|`) vs. Intersection (`&`) Types:** How do Union and Intersection operators work? If you intersect two object types (`TypeA & TypeB`), what happens to their properties? If you union them (`TypeA | TypeB`), why can you only access the properties that *both* types share without narrowing first?
5. **Generics (`<T>`):** Generics are often called "functions for types." How do they allow you to write reusable components (like a custom `useState` hook or an API fetcher) without losing type information? Explain the concept of a "Type Variable" and how it captures the type the user passes in.
6. **Type Narrowing & Type Guards:** How does TypeScript figure out what a variable's type is at a specific point in your code? Explain how standard checks (`typeof`, `instanceof`, `in`) narrow types. How do you write a **Custom Type Predicate** (e.g., `function isFish(pet: Fish | Bird): pet is Fish`) to teach the compiler about your own runtime logic?
7. **Literal Types & Discriminated Unions:** How do you restrict a string type to exact values (e.g., `type Role = "admin" | "user"`)? How do you combine literal types with **Discriminated Unions** (using a common property like `kind` or `type`) to create exhaustive `switch` statements that the compiler can verify?
8. **Enums vs. `as const` Objects:** Traditional TypeScript `enum` has several quirks (it generates runtime JavaScript code, numeric enums can be confusing). How does the modern pattern of using an object with the `as const` assertion (e.g., `const ROLES = { ADMIN: 'admin' } as const`) solve these issues and provide better type safety?
9. **Type Erasure & Runtime Reality:** "Types disappear at runtime." What does this mean for your compiled JavaScript? If you need to validate data coming from an external API (where TS types don't exist at runtime), how do you bridge the gap between compile-time types and runtime reality (e.g., using libraries like Zod or manual type guards)?
10. **The `tsconfig.json` & Strict Mode:** What is the purpose of the `tsconfig.json` file? Why is enabling `"strict": true` considered non-negotiable for professional projects? Explain what specific rules `strict` turns on (like `noImplicitAny` and `strictNullChecks`) and how they prevent entire categories of bugs.
