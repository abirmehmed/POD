# JavaScript Programming Language: Set 1
## The JavaScript Paradigm Shift: Asynchrony, Closures, and the Event Loop

## 📘 10 Conceptual Questions

1. **The Event Loop & Concurrency:** In languages like PHP or Python, code executes top-to-bottom and blocks on I/O. JavaScript is single-threaded but non-blocking. How does the Event Loop work? Explain the Call Stack, Web APIs, the Macrotask Queue (e.g., `setTimeout`), and the Microtask Queue (e.g., Promises). Why do Promises *always* execute before `setTimeout`, even if the timeout is set to `0`?
2. **Synchronous vs. Asynchronous Evolution:** What is the difference between blocking (synchronous) and non-blocking (asynchronous) execution? Trace the evolution of async JS: Callbacks ➔ Promises ➔ Async/Await. How did each step solve the specific problems of the previous one (e.g., escaping "Callback Hell", handling unhandled rejections)?
3. **Promises & States:** What is a Promise, and what are its three states (pending, fulfilled, rejected)? How does the `.then().catch().finally()` chain work under the hood? Explain why a Promise is "eager" (starts executing immediately upon creation) compared to a standard function.
4. **`async` / `await` & Syntactic Sugar:** How does `async/await` act as syntactic sugar over Promises? What actually happens to the execution thread when the engine hits an `await` keyword? How do you properly handle errors in an `async` function using `try/catch` compared to a standard Promise chain?
5. **Closures & Lexical Scope:** What is a closure, and how does it relate to lexical scoping? How can a function "remember" variables from its outer scope even after the outer function has returned and been garbage collected? Provide a conceptual explanation of how this enables data privacy and function factories.
6. **Scoping & Hoisting (`var` vs `let` vs `const`):** What are the fundamental differences between `var`, `let`, and `const` regarding function scope vs. block scope, hoisting, and the Temporal Dead Zone (TDZ)? Why does `const` prevent variable reassignment but *not* prevent the mutation of objects and arrays?
7. **The `this` Keyword & Context:** Unlike PHP or Java where `this` is strictly bound to the class instance, JS `this` is determined by *how a function is called*. Explain the four rules of `this` binding (default, implicit, explicit via `call`/`apply`/`bind`, and `new`). How do arrow functions break these rules by using lexical `this`?
8. **Prototypal Inheritance vs. Classical:** JavaScript uses prototypal inheritance, even when using the modern `class` keyword. How does the prototype chain work? When you access a property on an object, what is the exact lookup process up the chain, and how does it eventually resolve to `null`?
9. **Object References & Mutation:** JavaScript is strictly pass-by-value, but for objects and arrays, the "value" being passed is a memory reference. How does this lead to unexpected side effects when passing objects to functions or assigning them to new variables? How do shallow copies (spread operator) differ from deep clones (`structuredClone`)?
10. **Higher-Order Functions & Functional Paradigm:** What makes functions like `.map()`, `.filter()`, and `.reduce()` "higher-order"? How do they leverage closures and pure functions to process data declaratively? Why is this approach generally preferred over traditional `for` loops in modern JS development?
