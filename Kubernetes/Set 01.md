# Kubernetes Deep Dive: Set 1
## Core Primitives, Networking, Storage & The Control Plane

## 📘 10 Conceptual Questions
1. **The Pod Abstraction:** Why does Kubernetes wrap one or more containers in a "Pod" instead of just scheduling containers directly? Explain the shared network namespace (localhost) and shared storage volumes, and why this is critical for patterns like the "Sidecar" (e.g., Nginx + PHP-FPM).
2. **Deployments & ReplicaSets Under the Hood:** How does a Deployment actually execute a "Rolling Update"? Explain the mechanics of the `maxSurge` and `maxUnavailable` parameters, and how the Deployment controller manipulates the underlying ReplicaSet to achieve zero-downtime deployments.
3. **Service Types & Cluster DNS:** What is the exact difference between `ClusterIP`, `NodePort`, and `LoadBalancer`? How does CoreDNS resolve a service name (e.g., `mysql.default.svc.cluster.local`) to an IP address, and why should your Laravel app *never* connect to a hardcoded Pod IP?
4. **Ingress vs. LoadBalancer Service:** Both expose apps to the outside world, but how do they differ at the OSI model (Layer 4 vs Layer 7)? Why is an Ingress Controller (like Nginx or Traefik) required, and how does it handle TLS termination and host-based routing?
5. **ConfigMaps vs. Secrets:** How are these injected into Pods (Environment Variables vs. Volume Mounts)? Why are K8s Secrets *not* encrypted by default in `etcd`, and what is the mechanism to enable "Encryption at Rest" for the cluster?
6. **The Three Probes (Liveness, Readiness, Startup):** What is the specific purpose of each probe? Why is using a Liveness probe for a slow-starting PHP app a fatal mistake (causing CrashLoopBackOff), and how does the Startup probe solve this?
7. **Persistent Volumes (PV) vs. Persistent Volume Claims (PVC):** Explain the relationship between the physical storage (PV) and the application's request for storage (PVC). What are Access Modes (`ReadWriteOnce`, `ReadWriteMany`), and why does `ReadWriteMany` matter for Laravel's `storage/app/public` directory?
8. **Labels, Selectors & Annotations:** How do Labels and Selectors act as the "glue" that connects Services to Pods and Deployments to ReplicaSets? What is the difference between a Label (used for identification/selection) and an Annotation (used for metadata/tooling)?
9. **The Control Plane Components:** What are the specific roles of the `kube-apiserver`, `etcd`, `kube-scheduler`, and `kube-controller-manager`? Why is `etcd` considered the "single source of truth," and why must it be backed up religiously?
10. **Namespaces & Resource Quotas:** How do Namespaces provide logical isolation (e.g., `dev`, `staging`, `prod`)? How do Resource Quotas and LimitRanges prevent a single team or application from consuming all the CPU/Memory in a cluster?

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Local Cluster & Imperative vs. Declarative**<br>Install Minikube, Kind, or K3s. Create a Pod imperatively using `kubectl run`. Then, write the equivalent YAML file and apply it using `kubectl apply -f`. Delete the imperative pod. | You can seamlessly switch between `kubectl` commands for debugging and YAML files for actual deployment. |
| 2 | **Multi-Container Sidecar Pod**<br>Create a Pod with two containers: a basic PHP-FPM app and an Nginx reverse proxy. They must share an `emptyDir` volume for socket communication or static files. Expose only the Nginx container's port. | `kubectl logs <pod-name> -c nginx` works. The containers communicate over `localhost` inside the Pod. |
| 3 | **Zero-Downtime Rolling Update**<br>Deploy an app with `replicas: 3`. Update the image tag in the Deployment YAML. Watch `kubectl get pods` to see the new pods spin up and old pods terminate based on `maxSurge` and `maxUnavailable`. Intentionally deploy a broken image and execute `kubectl rollout undo`. | The app never experiences downtime during the update. The broken deployment is instantly reverted to the previous ReplicaSet. |
| 4 | **Internal Service Discovery**<br>Deploy a MySQL database and a PHP app. Create a `ClusterIP` service for MySQL. Exec into the PHP pod and use `curl` or `php -r` to ping the MySQL service using its DNS name (`mysql.default.svc.cluster.local`). | The PHP pod successfully resolves the DNS name and connects to the DB without knowing its IP address. |
| 5 | **Ingress Controller & Host Routing**<br>Install the Nginx Ingress Controller (via Helm or manifest). Create an Ingress resource that routes `multimart.local` to your PHP app service. Update your local `/etc/hosts` file and test in the browser. | Typing `multimart.local` in the browser loads the app. The Ingress controller logs show the routed request. |
| 6 | **ConfigMap & Secret Injection**<br>Create a ConfigMap for `APP_ENV` and a Secret for `DB_PASSWORD`. Mount the ConfigMap as Environment Variables, and mount the Secret as a Volume (file) in the Pod. Exec into the pod to verify both. | `env | grep APP_ENV` shows the value. `cat /etc/secrets/db_password` shows the password. |
| 7 | **Persistent Storage for Laravel**<br>Provision a PVC (e.g., 1Gi). Mount it to `/var/www/html/storage/app` in your PHP deployment. Upload a file via the app. Delete the Pod. Wait for the new Pod to start and verify the file is still there. | Data survives Pod destruction and recreation. The PVC status is `Bound`. |
| 8 | **Probe Tuning & Failure Simulation**<br>Add a Readiness probe to your app that checks `/health`. Intentionally make the `/health` route return a 500 error. Watch the Service endpoints. Fix the route and watch it rejoin the load balancer. | When the probe fails, `kubectl get endpoints` shows 0 ready addresses. Traffic stops routing to that pod immediately. |
| 9 | **Resource Limits & OOMKilled**<br>Set strict memory limits on a Pod (e.g., `limits: memory: 64Mi`). Write a quick PHP script that allocates 100MB of memory. Watch the Pod crash. Run `kubectl describe pod` to find the `OOMKilled` reason. | The Pod is killed by the kernel, not PHP. The `describe` output clearly shows the OOM event and the restart count increments. |
| 10| **Control Plane Inspection**<br>Run `kubectl get pods -n kube-system`. Identify the API server, etcd, and scheduler pods. Look at the logs of the `kube-scheduler` to see how it assigns a newly created Pod to a specific node. | You can visually identify the control plane components. You see the scheduler's decision-making process in the logs. |
