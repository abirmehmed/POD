# Kubernetes Deep Dive: Set 3
## Security, RBAC & Cluster Hardening

## 📘 10 Conceptual Questions
1. **ServiceAccounts vs. User Accounts:** What is the fundamental difference between a Kubernetes User (human or external system) and a ServiceAccount (Pod identity)? Why should every application Deployment have its own dedicated ServiceAccount with strictly scoped permissions, rather than using the `default` ServiceAccount?
2. **RBAC: Role vs. ClusterRole:** How do `Role` and `RoleBinding` differ from `ClusterRole` and `ClusterRoleBinding` in terms of scope? Why is granting `cluster-admin` to a CI/CD pipeline or a developer a catastrophic security anti-pattern?
3. **Pod Security Admission (PSA) / Standards:** Kubernetes replaced the deprecated `PodSecurityPolicy` with Pod Security Admission. Explain the three profiles: `Privileged`, `Baseline`, and `Restricted`. Why should all production workloads aim for the `Restricted` profile?
4. **Network Policies & Default Allow:** By default, Kubernetes operates on a "flat network" model where all Pods can talk to all other Pods. How do Network Policies change this to a Zero Trust model? Explain the difference between ingress (incoming) and egress (outgoing) rules.
5. **Secrets Management Reality Check:** Kubernetes `Secrets` are only base64 encoded, not encrypted by default. How do you secure them? Explain the progression from base64 -> etcd encryption at rest -> External Secrets Operator (integrating with AWS Secrets Manager, HashiCorp Vault, etc.).
6. **Security Contexts:** What do `runAsNonRoot`, `readOnlyRootFilesystem`, and `allowPrivilegeEscalation: false` actually do at the Linux kernel level? Why is running a PHP-FPM or Nginx container as `root` (UID 0) a massive vulnerability, even inside a container?
7. **Admission Controllers (Mutating vs. Validating):** How do Admission Controllers intercept requests to the API server before objects are persisted? Give examples of how tools like OPA Gatekeeper or Kyverno use Validating Admission to enforce organizational policies (e.g., "All images must come from our private registry").
8. **Namespace Isolation Limits:** While Namespaces provide logical separation (e.g., `dev` vs. `prod`), what do they *not* protect against by default? Why are Resource Quotas, Limit Ranges, and Network Policies required to achieve true multi-tenant isolation?
 a Cluster Compromise:** If an attacker gains shell access to a single Pod, what are their immediate next steps to escalate privileges to the Node or the Cluster? How do Security Contexts and RBAC limit this "blast radius"?
10. **Supply Chain Security:** How do you ensure that the container images running in your cluster haven't been tampered with? Explain the concepts of image signing (e.g., Cosign), vulnerability scanning (e.g., Trivy), and admission policies that block unsigned images.
