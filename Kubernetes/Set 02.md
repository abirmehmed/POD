# Kubernetes Deep Dive: Set 2
## Advanced Workloads, State & Auto-Scaling

## 📘 10 Conceptual Questions
1. **StatefulSets vs. Deployments:** Why can't you just use a Deployment for a database or a distributed cache like Redis? Explain the guarantees StatefulSets provide: stable, unique network identifiers, ordered graceful deployment/scaling, and stable persistent storage.
2. **Headless Services (`clusterIP: None`):** What is a headless service, and why is it the mandatory companion to a StatefulSet? How does it allow clients to discover and connect to individual Pods directly (e.g., `mysql-0.mysql.default.svc.cluster.local`) instead of load-balancing across them?
3. **Jobs & CronJobs at Scale:** How does Kubernetes guarantee a Job runs to completion? Explain `backoffLimit`, `activeDeadlineSeconds`, and `concurrencyPolicy` (Allow, Forbid, Replace). How do you map Laravel's `schedule:run` to a K8s CronJob effectively?
4. **Horizontal Pod Autoscaler (HPA):** How does the HPA controller decide to scale? Explain the difference between scaling on built-in metrics (CPU/Memory) via the Metrics Server vs. scaling on custom/external metrics (e.g., Redis queue length).
5. **KEDA (Kubernetes Event-driven Autoscaling):** Why is standard HPA insufficient for background workers? How does KEDA allow you to scale a Deployment from 0 to N replicas based on external event sources (like RabbitMQ, Kafka, or Redis streams), and what is "scale to zero"?
6. **Vertical Pod Autoscaler (VPA):** How does VPA differ from HPA? Explain how it analyzes historical resource usage and recommends (or automatically applies) new `requests` and `limits`. Why does applying VPA updates often require Pod restarts?
7. **Init Containers:** What is the specific lifecycle of an Init Container? Why is it the perfect pattern for running database migrations (`php artisan migrate`) or waiting for a dependency to be ready before the main application container starts?
8. **Pod Disruption Budgets (PDB):** During voluntary disruptions (like node upgrades or cluster autoscaling down), how does a PDB ensure high availability? Explain `minAvailable` vs. `maxUnavailable` and why every production Deployment should have one.
9. **DaemonSets:** What is the use case for a DaemonSet? Why is it the correct choice for node-level agents like log collectors (Fluentd), monitoring agents (Node Exporter), or CNI plugins, rather than a Deployment?
10. **Graceful Shutdown & `preStop` Hooks:** When K8s decides to terminate a Pod, it sends a `SIGTERM`. Why is this dangerous for a Laravel queue worker that might be halfway through processing a job? How do `terminationGracePeriodSeconds` and `preStop` hooks (e.g., `sleep 10` or `php artisan queue:restart`) prevent job loss?

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Deploy a StatefulSet**<br>Deploy a 2-replica Redis or MySQL cluster using a StatefulSet and a `volumeClaimTemplates`. Observe the Pod names (`app-0`, `app-1`). Delete `app-0` and verify the new pod retains the same name and reattaches to the same PVC. | Pod names are stable and predictable. Persistent data survives Pod deletion and recreation. |
| 2 | **Headless Service Discovery**<br>Create a `clusterIP: None` Service for your StatefulSet. Exec into another Pod and run `nslookup <service-name>`. Verify it returns the individual IP addresses of the StatefulSet Pods, not a single virtual IP. | DNS query returns multiple A records (one for each Pod). Direct Pod-to-Pod communication is possible using stable DNS names. |
| 3 | **Laravel Scheduler CronJob**<br>Create a K8s CronJob that runs `php artisan schedule:run` every minute. Set `concurrencyPolicy: Forbid` and `successfulJobsHistoryLimit: 3`. Verify it executes and cleans up old Job pods automatically. | CronJob triggers reliably. Old completed/failed Job pods are garbage collected, preventing cluster clutter. |
| 4 | **CPU-Based HPA Implementation**<br>Deploy an app with HPA configured to scale between 2 and 5 replicas at 50% CPU utilization. Use a load-testing tool (like `hey`) to spike CPU. Watch `kubectl get hpa` and `kubectl get pods` to see it scale up, then scale down when traffic stops. | HPA successfully scales out under load and scales in when idle. Metrics Server is correctly providing data. |
| 5 | **Queue Worker Auto-Scaling (KEDA or Custom Metrics)**<br>*(Advanced)* Install KEDA (or use a custom metrics adapter). Configure a scaler that monitors a Redis list length. Push 1,000 dummy jobs to the queue. Watch the queue worker Deployment scale from 1 to N replicas to drain the queue, then scale back to 1 (or 0). | Worker replicas increase proportionally to queue depth. Queue is processed faster due to parallel workers. Scales down when queue is empty. |
| 6 | **Init Container for Migrations**<br>Add an Init Container to your main app Deployment that runs `php artisan migrate --force`. Use a simple shell script or `wait-for-it` to ensure the DB is reachable before running the migration. | The main PHP-FPM container remains in `Init:0/1` state until the migration Init Container exits with code 0. |
| 7 | **Pod Disruption Budget (PDB) Test**<br>Apply a PDB with `minAvailable: 2` to a 3-replica Deployment. Attempt to `kubectl drain` the node hosting two of those pods. Observe how K8s blocks the eviction of the second pod to respect the budget. | `kubectl drain` hangs or reports a PDB violation, proving that K8s will not sacrifice application availability for node maintenance. |
| 8 | **DaemonSet Log Forwarder**<br>Deploy a simple DaemonSet (e.g., a basic `busybox` loop that echoes to a file, or a real `node-exporter`). Verify that exactly one Pod is running on *every* node in your cluster, including newly added nodes. | `kubectl get pods -o wide` shows one instance of the DaemonSet Pod scheduled on every available node. |
| 9 | **Graceful Queue Worker Shutdown**<br>Configure your queue worker Deployment with `terminationGracePeriodSeconds: 30`. Add a `preStop` hook that runs `php artisan queue:restart` or `sleep 15`. Simulate a pod deletion and verify the worker finishes its current job before exiting. | Pod enters `Terminating` state. The worker does not abruptly die mid-job. The pod exits cleanly with code 0 after the grace period. |
| 10| **VPA Recommendation Analysis**<br>Install the VPA CRDs. Create a VPA resource in `Auto` or `Recommendation` mode for your app. Let it run under normal load for a while. Check the VPA status to see its recommended CPU/Memory requests vs. what you actually configured. | VPA output shows a clear recommendation (e.g., "Current request: 100m, Recommended: 45m"). You understand how to right-size based on actual telemetry. |
