# Kubernetes Deep Dive: Set 4 (Final)
## Helm, GitOps, Operators & Day-2 Operations

## 📘 10 Conceptual Questions
1. **Helm Architecture & Templating:** How does Helm solve the "YAML sprawl" problem? Explain the relationship between a `Chart`, `values.yaml`, and the rendered Kubernetes manifests. Why is Helm often called the "package manager for Kubernetes"?
2. **GitOps Philosophy (Pull vs. Push):** Traditional CI/CD *pushes* changes to a cluster (e.g., `kubectl apply` from Jenkins). How does GitOps (e.g., ArgoCD, Flux) invert this model? Why is having an agent *pull* the desired state from a Git repository considered more secure and reliable for cluster management?
3. **Helm vs. Kustomize:** Both manage Kubernetes configurations, but they solve different problems. Why is Helm better suited for packaging and distributing applications with variables, while Kustomize is often preferred for environment-specific overlays (dev/staging/prod) without templating engines?
4. **The GitOps Secret Paradox:** Git repositories should not contain plaintext secrets. How do GitOps workflows handle Kubernetes Secrets? Explain solutions like Mozilla SOPS, Bitnami Sealed Secrets, or the External Secrets Operator.
5. **Custom Resource Definitions (CRDs) & Operators:** How do CRDs extend the Kubernetes API? What is the "Operator Pattern" (Control Loop), and how does it automate Day-2 operations (like database backups, upgrades, or scaling) that a standard Deployment cannot handle?

7. **Day-2 Cluster Upgrades:** What is the safe, step-by-step process for upgrading a Kubernetes cluster's control plane and worker nodes? How do Pod Disruption Budgets (PDBs) and node draining ensure zero downtime for applications during this process?
8. **Backup & Disaster Recovery (Velero):** What is the difference between backing up `etcd` (cluster state) and backing up Persistent Volumes (application data)? How does a tool like Velero coordinate both to provide a complete cluster restore capability?
9. **Helm Hooks & Lifecycle Management:** How do Helm annotations (e.g., `helm.sh/hook: pre-install`, `post-upgrade`) allow you to run specific Jobs (like database migrations or cache warming) at precise moments during a release lifecycle?
10. **Drift Detection & Self-Healing:** If a developer manually runs `kubectl edit deployment` to change an image tag, how does a GitOps controller detect this "drift"? What is the auto-sync process, and why is it critical for maintaining cluster integrity?
