# System Design Case Studies: Set 3
## Social Media Feed: Twitter / Instagram / Facebook

## 📘 10 Conceptual Questions
1. **Fan-Out on Write vs. Fan-Out on Read:** What is the fundamental trade-off between pushing a new post to all followers' feeds immediately (Fan-out on Write) vs. fetching and merging posts from followed users when the feed is requested (Fan-out on Read)? Which is better for read-heavy vs. write-heavy users?
2. **The "Justin Bieber" Problem (Hybrid Approach):** If a user with 100 million followers posts, Fan-out on Write will overwhelm the system. How do you design a hybrid system that uses Fan-out on Write for normal users, but Fan-out on Read (or a separate celebrity cache) for high-follower accounts?
3. **Database Selection for Feeds:** Why are wide-column NoSQL databases (like Cassandra or ScyllaDB) often preferred for storing user timelines over traditional relational databases? How does the partition key (e.g., `user_id`) optimize the "get my feed" query?
4. **Distributed Counters at Scale:** How do you accurately count millions of likes, retweets, or views per second without creating a database write bottleneck? Explain the use of in-memory counters (Redis `INCR`) and periodic asynchronous flushing to persistent storage.
5. **Cursor-Based Pagination for Infinite Scroll:** Why is `LIMIT 50 OFFSET 10000` a performance anti-pattern for social media feeds? How do you use a `cursor` (e.g., the ID or timestamp of the last item) to fetch the next page efficiently and consistently, even as new posts are added?
6. **Feed Ranking & Scoring Algorithms:** A chronological feed is simple, but modern apps use ranked feeds. How do you calculate a "relevance score" (combining recency, engagement, and user affinity)? Should this ranking happen in real-time during the read, or pre-computed in a batch pipeline?
7. **Graph Data Modeling for Relationships:** How do you efficiently store and query "follower/following" relationships? Compare using an Adjacency List in a relational DB vs. Redis Sets vs. a dedicated Graph Database (like Neo4j).
8. **Media Delivery & CDN Architecture:** Why should user-uploaded images/videos never be served directly from your application servers? Explain the flow of resizing, optimizing, caching at the Edge (CDN), and serving via adaptive bitrate streaming for video.
9. **Cache Invalidation & Edge Cases:** If a user deletes a post, blocks another user, or makes their account private, how do you ensure that post disappears from the cached feeds of their followers without rebuilding the entire feed from scratch?
10. **Idempotency in Social Actions:** How do you prevent a user from accidentally "liking" a post twice due to a network retry or double-tap? Explain the use of idempotency keys or unique composite indexes (`user_id`, `post_id`) in the database.

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Fan-Out on Write Implementation**<br>Build a background job that triggers when a user creates a post. The job fetches all follower IDs and inserts a reference to the new `post_id` into each follower's `feed_timeline` table/collection. | Post appears instantly in followers' feeds. Write load is handled asynchronously via queues, keeping the post-creation API fast. |
| 2 | **Hybrid Feed Logic (The Celebrity Fix)**<br>Modify the feed generation logic: if the author has > 10,000 followers, *do not* fan-out on write. Instead, flag the post as "celebrity". During feed read, merge the user's pushed feed with a real-time query of celebrity posts they follow. | Normal users get instant push feeds. Celebrity posts are fetched on read, preventing massive write storms. |
| 3 | **Timeline Database Schema**<br>Design a wide-column or optimized SQL schema for a `user_feed` table. Primary key: `(user_id, post_timestamp DESC)`. Write the query to fetch the first 20 items for a specific user. | Query uses the composite primary key for a highly efficient index scan. No sorting (`ORDER BY`) is required at query time. |
| 4 | **Distributed Like Counter**<br>Implement a Redis-based counter for post likes. On "like", `INCR` the Redis key. Create a scheduled Laravel command that runs every 5 minutes to sync the Redis counts to the persistent database. | Likes register instantly with zero DB writes. The background sync accurately persists the data without losing counts. |
| 5 | **Cursor Pagination API**<br>Build `GET /api/feed?cursor={last_post_id}`. The query fetches posts `WHERE user_id = X AND post_id < {last_post_id} ORDER BY post_id DESC LIMIT 20`. Return the next cursor in the response metadata. | API returns consistent results even if new posts are added during the request. No `OFFSET` is used. Response includes `next_cursor`. |
| 6 | **Follow/Unfollow with Redis Sets**<br>Use Redis Sets to manage follows: `SADD followers:{user_id} {follower_id}`. Write a query to find "Mutual Followers" between two users using `SINTER`. | Follow/unfollow operations are O(1). Mutual follower lookup is lightning fast and executed entirely in memory. |
| 7 | **Basic Feed Ranking Algorithm**<br>Create a scoring function for posts: `Score = (Recency_Decay) + (Likes * 2) + (Comments * 5)`. Sort a fetched batch of posts by this score before returning them to the client. | Feed is no longer strictly chronological. High-engagement, recent posts appear at the top. Scoring logic is isolated and testable. |
| 8 | **Direct-to-S3 Media Upload**<br>Implement the presigned URL flow for post images. Backend generates URL → Frontend uploads directly to S3 → Frontend sends post creation request with the S3 URL. | Application server CPU/Network remains flat during large uploads. Images are immediately available via CDN. |
| 9 | **Soft Delete & Block Handling**<br>When a user is blocked, write a script or job that scans the blocker's feed and filters out (or soft-deletes) references to the blocked user's posts. | Blocked user's content disappears from the feed. The solution handles the edge case without requiring a full feed rebuild. |
| 10| **Feed Load Testing**<br>Use a tool like `k6` to simulate 500 concurrent users requesting their feeds. Monitor the database CPU, Redis memory, and API response times. Identify and fix the first bottleneck. | 95th percentile response time remains < 200ms. Database does not spike to 100% CPU. Caching is effectively reducing DB load. |
