# System Design Case Studies: Set 1
## The Foundations: URL Shortener & Distributed Rate Limiter

## 📘 10 Conceptual Questions
1. **The Read/Write Ratio Reality:** A URL shortener typically has a 100:1 read-to-write ratio. How does this specific ratio dictate your entire architecture, from database choice (SQL vs. NoSQL) to caching strategies and load balancing?
2. **Hash Generation & Collisions:** How do you generate a short URL (e.g., 7-character Base62)? What is the mathematical probability of a collision, and what is the most efficient algorithmic strategy to resolve one when it inevitably happens?
3. **Distributed Unique ID Generation:** If you move away from simple hashing to pre-generated unique IDs (to avoid collision checks), how do services like Twitter Snowflake or Flickr's ticket servers generate globally unique, time-sortable IDs across multiple nodes without a central bottleneck?
 in Memory (RAM) vs. Persistent Storage:** Why is Redis/Memcached absolutely critical for a URL shortener? Explain the "Cache-Aside" pattern here, and how you prevent a "thundering herd" if a viral URL's cache key expires.
5. **NoSQL vs. Relational for this Use Case:** Why might a key-value store (like DynamoDB or Cassandra) be a better fit than PostgreSQL for the core mapping table? When would you *still* choose a relational database (e.g., for complex analytics or user accounts)?
6. **Analytics & Asynchronous Processing:** How do you track click analytics (geo-location, device, timestamp) without adding latency to the redirect? Explain how to use message queues to decouple the fast redirect from the slow analytics write.
7. **Rate Limiting at the Edge:** How do you prevent a single user or bot from generating 10,000 short URLs per minute? Compare token bucket vs. sliding window algorithms, and explain why this check should happen at the API Gateway/Edge, not the application layer.
8. **Garbage Collection & Data Lifecycle:** How do you handle URL expiration? Compare the performance impact of a cron job scanning for expired rows vs. a database TTL (Time-To-Live) feature vs. lazy deletion on read.
9. **Custom Aliases & Vanity URLs:** How does allowing users to choose their own short code (e.g., `short.ly/my-brand`) change the system design? How do you prevent squatting and ensure fast lookups for both hashed and custom URLs?
10. **API Design & Idempotency:** How should the REST API for creating a short URL be designed? Why is it important for the creation endpoint to be idempotent (returning the same short URL if the same long URL is submitted twice)?
