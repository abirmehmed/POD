# System Design Case Studies: Set 4 
## High-Concurrency E-Commerce: Flash Sales & Inventory Management

## 📘 10 Conceptual Questions
1. **The Overselling Problem (Race Conditions):** Why does a simple `if (stock > 0) { stock--; save(); }` fail catastrophically under high concurrency? Explain the "Lost Update" anomaly and how database isolation levels play a role.
2. **Pessimistic vs. Optimistic Locking at Scale:** When should you use `SELECT ... FOR UPDATE` (pessimistic) vs. a `version` column (optimistic) vs. Redis `DECR` for inventory management during a flash sale? What are the performance trade-offs of each?
3. **The Virtual Waiting Room:** How do you protect your database from being crushed by 100,000 concurrent requests at exactly 12:00:00 PM? Explain token-based queuing (e.g., Cloudflare Waiting Room or a custom Redis queue) that throttles traffic *before* it hits your application.
4. **Idempotency in Checkout:** Why is an idempotency key absolutely critical for the "Place Order" API endpoint during a flash sale? How does it prevent double-charging or duplicate orders if the client's network drops and automatically retries?
5. **Inventory Reservation & TTL:** How do you handle "abandoned carts" during a high-demand sale? Explain the pattern of temporarily reserving inventory in Redis with a strict TTL (e.g., 10 minutes) and releasing it back to the pool if payment isn't completed.
6. **Asynchronous Order Processing:** Why should the "Order Placed" response be returned immediately after reserving inventory, while payment processing, email notifications, and warehouse alerts happen asynchronously via a message queue?
7. **Database Hotspots & Inventory Sharding:** If a single "iPhone 15" database row is being updated 10,000 times a second, it becomes a write bottleneck (row lock contention). How do you shard this inventory (e.g., splitting stock into 100 sub-rows) to distribute the write load?
8. **Cache Stampede Prevention:** If 50,000 users hit the product page simultaneously, how do you serve the page without crashing the DB? Explain request coalescing (mutex locks) and the acceptability of serving slightly stale cache data during peak traffic.
9. **Approximate Counting (HyperLogLog):** How do you update the "1,240 people are viewing this" or "Only 5 left!" counters in real-time without hammering the database with individual `INSERT` or `UPDATE` statements?
10. **Circuit Breakers & Graceful Degradation:** If the external payment gateway slows down to a 5-second response time during the sale, how do you prevent your entire order processing queue from backing up, exhausting database connections, and taking down the whole site?

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Optimistic Concurrency Control**<br>Add a `version` column to the `products` table. Implement a checkout flow that checks the version on update. Simulate 10 concurrent requests for 1 item. Verify 1 succeeds and 9 fail gracefully with a `StaleObjectException`. | No overselling occurs. Failed requests return a clear "Item no longer available" message to the user. |
| 2 | **Pessimistic Locking (`FOR UPDATE`)**<br>Implement a checkout flow using Laravel's `lockForUpdate()`. Test with a concurrent script (e.g., 50 simultaneous requests for 10 items). Verify exactly 10 succeed, stock hits 0, and no negative stock is possible. | Database enforces strict sequential access. Zero race conditions. Query logs confirm `FOR UPDATE` is used. |
| 3 | **Redis Inventory Reservation (TTL)**<br>Build a flow where clicking "Buy" decrements a Redis counter and sets a 10-minute TTL key for that `user_id` + `product_id`. Write a script that scans for expired keys and increments the counter back. | Inventory is instantly reserved. Abandoned reservations auto-release after 10 minutes. No DB writes occur during reservation. |
| 4 | **Idempotency Middleware**<br>Create middleware that checks an `Idempotency-Key` header. Simulate a network timeout and retry. Verify the second request returns the cached success response (e.g., the original Order ID) without re-processing the payment or DB insert. | Duplicate requests are intercepted in <5ms. No duplicate database records or external API calls are made. |
| 5 | **Virtual Waiting Room Simulation**<br>Build a middleware that checks a Redis queue capacity. If full, return a "Please wait, you are in line" HTML page with JS polling. If not full, allow the request and add a temporary token. | Application server is protected from sudden traffic spikes. Users are gracefully queued instead of seeing 502/503 errors. |
| 6 | **Asynchronous Order Fulfillment**<br>Refactor the checkout controller to dispatch a `ProcessOrderJob` to a high-priority queue immediately after DB reservation, returning a `202 Accepted` response to the user. | Controller response time is <50ms. Payment and emails are processed in the background. User sees a "Processing" UI state. |
| 7 | **Inventory Sharding (Sub-rows)**<br>Split a product's stock of 10,000 into 10 rows of 1,000 (`product_id_1`, `product_id_2`, etc.). Write a service that randomly picks an available sub-row to decrement, distributing the write lock contention. | Write load is distributed across multiple rows. Concurrent checkout tests show significantly fewer lock timeouts compared to a single row. |
| 8 | **HyperLogLog for "Viewers Count"**<br>Implement a Redis HyperLogLog (`PFADD`) to track unique visitors viewing a product page per minute. Display "X people are viewing this" without storing individual user IDs or IPs. | Memory usage remains constant (~12KB) regardless of whether 100 or 1,000,000 users view the page. Count is accurate within a ~2% margin of error. |
| 9 | **Circuit Breaker for Payment Gateway**<br>Wrap the external payment API call in a circuit breaker. Simulate a 5-second timeout. Verify that after 5 failures, the circuit opens, and orders are queued for later retry instead of hanging the worker. | Circuit opens correctly. Fallback/queueing activates. System recovers automatically when the mock API starts responding again. |
| 10| **Flash Sale Load Test & Bottleneck ID**<br>Use `k6` or `hey` to blast the checkout endpoint with 300+ RPS. Monitor DB connections, Redis memory, and queue depth. Document the first point of failure and implement one fix to resolve it. | Bottleneck is clearly identified (e.g., DB max connections, queue backup). Fix is implemented and verified to increase throughput. |
