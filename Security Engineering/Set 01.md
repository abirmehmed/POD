# Security Engineering: Set 1
## Application Security & OWASP Top 10 Deep Dive

## 📘 10 Conceptual Questions
1. **SQL Injection (SQLi) Mechanics:** How does SQLi actually work at the database parser level? Why do prepared statements (parameterized queries) completely neutralize this attack, and when does Laravel's Query Builder or Eloquent *still* leave you vulnerable (e.g., raw expressions, column name injection)?
2. **Cross-Site Scripting (XSS) Vectors:** What is the difference between Reflected, Stored, and DOM-based XSS? Why is `htmlspecialchars()` not always enough, and how do modern frameworks like Laravel (via Blade) automatically mitigate this? What are the edge cases (e.g., injecting into `<script>` tags or `href` attributes)?
3. **Cross-Site Request Forgery (CSRF):** How does a CSRF attack trick a user's browser into executing an unwanted action? Explain the mechanics of the Synchronizer Token Pattern (which Laravel uses) and why `SameSite` cookie attributes are a crucial secondary defense.
4. **Mass Assignment Vulnerabilities:** What is the danger of blindly passing `$request->all()` to a model's `create()` or `update()` method? How do Laravel's `$fillable` and `$guarded` properties work, and why is `$guarded = []` a massive security risk?
5. **Server-Side Request Forgery (SSRF):** How does an attacker trick your server into making HTTP requests to internal resources (e.g., `http://169.254.169.254` for AWS metadata)? How do you validate and sanitize user-provided URLs before making cURL/Guzzle requests?
6. **Insecure Direct Object Reference (IDOR):** What is IDOR, and why is it consistently in the OWASP Top 10? How does it differ from a general authorization failure, and why are Laravel Policies the most robust way to prevent it?
7. **Security Misconfiguration:** Why are default passwords, verbose error messages (e.g., `APP_DEBUG=true` in production), and missing HTTP security headers considered critical vulnerabilities? How does information leakage aid attackers in the reconnaissance phase?
8. **XML External Entity (XXE) Injection:** How does parsing untrusted XML data allow an attacker to read local files or perform SSRF? Why is disabling `libxml_disable_entity_loader` (or using safe parsers) mandatory when handling XML in PHP?
9. **Deserialization Vulnerabilities:** How can unserializing untrusted user input (e.g., PHP's `unserialize()`, or insecure JSON parsing) lead to Remote Code Execution (RCE) via "Magic Methods" (`__wakeup`, `__destruct`)? Why is JSON generally safer than PHP serialization?
10. **Security Defense in Depth:** Why is relying on a single security control (e.g., "I have a WAF") insufficient? Explain how layering controls (Input Validation → Output Encoding → Parameterized Queries → Least Privilege) creates a resilient system.

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Mass Assignment Audit & Fix**<br>Search your codebase for `Model::create($request->all())` or `update()`. Refactor them to use Form Requests with strict `rules()` and explicit field mapping, or strictly defined `$fillable` arrays. | Zero instances of `$request->all()` in mass assignment. Static analysis (Larastan) confirms no unexpected properties are being set. |
| 2 | **IDOR Prevention via Policies**<br>Create an endpoint `GET /api/orders/{id}`. Write a test where User A tries to access User B's order. Implement a Laravel Policy (`OrderPolicy::view`) to enforce ownership. | Test passes: User A gets `403 Forbidden`. User B gets `200 OK`. Policy logic is centralized and reusable. |
| 3 | **Custom XSS Sanitization**<br>Create a scenario where users can input "safe" HTML (e.g., bold/italic). Use a library like `mews/purifier` (HTMLPurifier) to sanitize the input before storing, and verify that `<script>` tags are stripped while `<b>` tags remain. | Stored payload `<script>alert(1)</script><b>hello</b>` renders safely as `<b>hello</b>` in the browser. No JS executes. |
| 4 | **SSRF Protection Implementation**<br>Build a "Fetch URL Metadata" feature. Implement a validation rule that rejects private IP ranges (10.x.x.x, 192.168.x.x, 127.x.x.x) and localhost. Test with a tool like `ssrf-server` or manual curl. | Requests to internal IPs are blocked with a `400 Bad Request`. Only public, valid HTTP/HTTPS URLs are allowed. |
| 5 | **CSRF Token Enforcement Test**<br>Create a simple HTML form outside your Laravel app that attempts to POST to a protected Laravel route. Verify it fails with a `419 Page Expired` (TokenMismatchException). Then, properly include the `@csrf` token and verify it succeeds. | External POST requests are reliably blocked. Legitimate internal requests with valid tokens succeed. |
| 6 | **Secure Error Handling**<br>Configure `APP_DEBUG=false` in `.env`. Trigger a 500 error. Verify the user sees a generic `500.blade.php` view, while the actual stack trace is logged securely to `storage/logs/laravel.log` (or an external service). | No stack traces, database credentials, or environment variables are exposed in the HTTP response. |
| 7 | **Rate Limiting Abuse Prevention**<br>Apply Laravel's `throttle` middleware to a sensitive endpoint (e.g., password reset or login). Configure it to `throttle:5,1` (5 attempts per minute). Test with a script sending 10 rapid requests. | First 5 requests succeed. Requests 6-10 return `429 Too Many Requests` with a `Retry-After` header. |
| 8 | **Secure File Upload Validation**<br>Build a file upload endpoint. Validate not just the extension, but the MIME type (using `mimes` and `mimetypes` rules), file size, and rename the file to a random UUID before storing it outside the public directory. | Uploading a `.php` file disguised as `.jpg` is rejected. Stored files cannot be executed by the web server. |
| 9 | **Dependency Vulnerability Scanning**<br>Run `composer audit` and `npm audit` on your project. Identify any packages with known CVEs. Update or replace the vulnerable packages and verify the audit passes. | Zero high/critical vulnerabilities reported. `composer.lock` and `package-lock.json` are updated and committed. |
| 10| **Security Headers Middleware**<br>Create a middleware that injects strict security headers: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and a basic `Content-Security-Policy`. | Running a site through Mozilla Observatory or SecurityHeaders.com yields an A or A- grade. |
