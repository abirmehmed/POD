# Security Engineering: Set 3
## Infrastructure, Network & Browser Security

## 📘 10 Conceptual Questions
1. **TLS/SSL Mechanics & The Handshake:** How does the TLS handshake actually establish a secure, encrypted channel? Why is "HTTPS everywhere" non-negotiable, and what happens if a certificate expires or is issued by an untrusted Certificate Authority (CA)?
2. **HTTP Strict Transport Security (HSTS):** What is HSTS, and how does it prevent protocol downgrade attacks and cookie hijacking? Why is submitting your domain to the "HSTS Preload List" baked into browsers the ultimate defense against first-visit HTTP attacks?
3. **Content Security Policy (CSP) Deep Dive:** How does CSP mitigate XSS and data injection attacks? Why are `nonce` or `hash`-based CSPs vastly superior to `unsafe-inline`, and how do you safely allow third-party scripts (like Stripe or Google Analytics) without opening security holes?
4. **CORS vs. CSRF:** Why is Cross-Origin Resource Sharing (CORS) *not* a security mechanism against Cross-Site Request Forgery (CSRF)? Explain how CORS only restricts the *browser's ability to read the response*, not the server's obligation to process the request.
5. **Web Application Firewalls (WAF):** How do WAFs (like Cloudflare, AWS WAF, or ModSecurity) detect malicious traffic? Why is a WAF considered a "safety net" and not a replacement for secure application code (Defense in Depth)?
6. **Zero Trust Network Architecture (ZTNA):** What does "Never trust, always verify" mean in practice? How does micro-segmentation (e.g., ensuring the database only accepts connections from the application server's specific IP/security group) reduce the blast radius of a compromise?
7. **Secure Cookie Attributes:** Explain the exact purpose of `Secure`, `HttpOnly`, `SameSite` (Strict vs. Lax), `Domain`, and `Path`. How does combining these attributes create a robust defense against XSS and CSRF?
8. **Subresource Integrity (SRI):** How does SRI protect your application if a third-party CDN (like cdnjs or a compromised npm package) serves a malicious version of a JavaScript file? Explain the role of cryptographic hashes in the `<script>` tag.
9. **DDoS Mitigation (Layer 3/4 vs. Layer 7):** What is the difference between a volumetric attack (flooding bandwidth) and an application-layer attack (e.g., slowloris, or hitting your most expensive database query 10,000 times/sec)? How do you mitigate each?
10. **DNS Security (DNSSEC & CAA):** How does DNSSEC prevent DNS spoofing/cache poisoning? What is a CAA (Certificate Authority Authorization) record, and how does it prevent an attacker from tricking a rogue CA into issuing a valid SSL certificate for your domain?

## 🛠️ 10 Practical Tasks

| # | Task | Success Criteria |
|---|------|------------------|
| 1 | **Enforce HTTPS & HSTS**<br>Configure your web server (Nginx/Apache) or Laravel middleware to redirect all HTTP traffic to HTTPS. Add the `Strict-Transport-Security` header with `max-age=31536000; includeSubDomains; preload`. | HTTP requests return a 301 redirect to HTTPS. Response headers include the exact HSTS string. |
| 2 | **Implement Strict CSP**<br>Add a `Content-Security-Policy` header. Start with `default-src 'self'`. Add specific allowances for required external resources (e.g., `script-src 'self' https://js.stripe.com`). Remove all `unsafe-inline` and `unsafe-eval`. | Browser console shows no CSP violations. External scripts load correctly. Inline scripts are blocked unless they use a valid `nonce`. |
| 3 | **Secure CORS Configuration**<br>Configure Laravel's `cors.php` to explicitly define `allowed_origins`, `allowed_methods`, and `allowed_headers`. Ensure `supports_credentials` is true if using cookies, and test with a cross-origin `fetch` request. | Requests from unauthorized origins are blocked by the browser. Authorized origins receive the correct `Access-Control-Allow-*` headers. |
| 4 | **Fortify Session Cookies**<br>Audit Laravel's `config/session.php`. Ensure `secure => env('SESSION_SECURE_COOKIE', true)`, `http_only => true`, and `same_site => 'lax'` (or `'strict'`). Verify the `Set-Cookie` header in the browser's Network tab. | Cookies are invisible to JavaScript (`HttpOnly`), only sent over HTTPS (`Secure`), and protected from cross-site context (`SameSite`). |
| 5 | **Subresource Integrity (SRI) Implementation**<br>Include a third-party library (e.g., Alpine.js or Tailwind CDN) in your Blade template. Generate its SHA-384 hash and add the `integrity` and `crossorigin="anonymous"` attributes to the `<script>` or `<link>` tag. | Browser loads the script. If you manually alter the hash in the HTML, the browser blocks execution and logs an SRI error in the console. |
| 6 | **Edge Rate Limiting (Nginx/Cloudflare)**<br>Configure a basic rate limit at the web server level (e.g., Nginx `limit_req_zone`) or via Cloudflare WAF rules to block IPs making >10 requests per second to `/login`. | Rapid, automated requests receive a `429 Too Many Requests` or `503 Service Unavailable` directly from the edge, never reaching PHP. |
| 7 | **Network Segmentation (Docker)**<br>Set up a `docker-compose.yml` with three services: `app`, `db`, and `redis`. Put `db` and `redis` on an internal-only network. Ensure the `app` can reach them, but a separate "attacker" container on the default network cannot ping or connect to the DB port. | Internal services are completely isolated from external network access. Only the app container can resolve and connect to the database. |
| 8 | **Security Headers Audit**<br>Deploy your app to a public URL (or use a tool like ngrok). Run it through [SecurityHeaders.com](https://securityheaders.com) or Mozilla Observatory. Tweak your headers until you achieve an "A" or "A+" grade. | Report shows A/A+ grade. Missing headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy) are all implemented. |
| 9 | **DNS CAA Record Setup**<br>Go to your domain's DNS provider (e.g., Cloudflare, Route53). Add a CAA record that explicitly allows only your chosen CA (e.g., `0 issue "letsencrypt.org"`). Verify using a DNS lookup tool. | DNS query for CAA records returns only your specified CA. Rogue CAs are cryptographically prevented from issuing certs for your domain. |
| 10| **WAF Rule Simulation**<br>Write a simple Nginx config or middleware that inspects the request URI/query string for common SQLi patterns (e.g., `' OR 1=1`, `UNION SELECT`). Return a `403 Forbidden` if matched. | Sending a URL like `/?id=1' OR 1=1--` is instantly blocked with a 403, before the request ever hits Laravel's routing. |
